# Conversational Operator Confirmation Core Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Make every chat-originated strategy analysis/research request produce an auditable action proposal and require a separate explicit confirmation before any task is created or queued.

**Architecture:** Keep the existing classifier and deterministic guard, but change task-producing guard decisions into immutable proposals persisted behind a new hexagonal repository. `handleChatMessage` resolves a pending confirmation before calling the classifier; a successful compare-and-set confirmation executes the stored task snapshot through the existing task-intake chokepoint. This first slice deliberately excludes similarity/RAG, bot catalog resolution, and office completion summaries.

**Tech Stack:** TypeScript ESM, Zod, Hono, Drizzle ORM/Postgres, BullMQ port, Vitest.

---

## Scope Boundaries

This plan implements delivery slice 1 from the approved design:

- persisted `ActionProposal` with exactly-once confirmation state transition;
- typed `pendingInteraction` in chat session state;
- deterministic confirm/cancel resolver before LLM classification;
- `assistant_message` response with structured confirm/cancel actions;
- strategy onboarding and onboard-then-research requests require confirmation;
- audit events and E2E proof that no queue write occurs before confirmation.

Separate plans will cover:

- exact-match enrichment and hybrid strategy similarity;
- meaningful analyst/research completion replies;
- bot catalog and entity disambiguation;
- research-artifact RAG and reranking.

- [ ] **Step 3: Add the pre-classifier confirmation branch**

At the start of `handleChatMessage`, before `chat.intent_classifier.started`:

1. Check `input.session.pendingInteraction`.
2. Resolve the reply with `resolveConfirmationReply`.
3. For cancel, call `cancelPending`, clear session state, emit `chat.proposal.cancelled`, return `assistant_message`.
4. For unresolved, return `assistant_message` with the same confirm/cancel actions and do not call the classifier.
5. For confirm, call `confirmPending` using the current session ID and current time.
6. Execute only `confirmed_now`; call `createAndEnqueueTask` with the stored task snapshot and stored source.
7. Create the existing `ChatPlan` from the stored chain after intake succeeds.
8. Call `attachTask`, clear pending interaction, update session task/goal pointers, emit `chat.proposal.confirmed` followed by `chat.task_created`, and return `task_created`.
9. For `already_confirmed`, return the attached task status when `confirmedTaskId` exists; otherwise return an `assistant_message` saying confirmation is already processing. Never enqueue from this branch.
10. For expired/not-found, clear the pending interaction and return a specific `assistant_message` without enqueueing.

Keep task intake in one private helper so normal confirmed execution and chain creation cannot diverge from existing behavior.
