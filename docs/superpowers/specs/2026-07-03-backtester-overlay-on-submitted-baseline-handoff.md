# Platform handoff prompt — native overlay-on-submitted-baseline + multi-overlay bundles

## Why

G3 introduced `strategy_revision`: a deterministic, lab-side merge of proxy-passed hypothesis overlays into one composed strategy. Today the composition happens **entirely in trading-lab** (`src/research/compose-revision-bundle.ts`) — the base strategy's `onBarClose` and every included overlay's `apply` are namespace-isolated into a single self-contained module, then submitted as ONE `kind:'strategy'` bundle via the existing `engine:'strategy'` lane (G1). This works, but it means:

- the backtester never sees the revision as "baseline + overlays" — it only ever sees one opaque composed module;
- lab must re-implement (and keep in lockstep with) the engine's own `OverlayComposer` veto/patch/annotate/pass semantics in TypeScript templating (`compose-revision-bundle.ts`'s header comment documents this duplication in detail, including a **confirmed** gap: the engine has no rule-string interpreter, so Style-A data-only overlays are `unsupported_module_shape`, not a lab shortcut);
- there is a **pre-existing mismatch** the G3 spec calls out in its own §0: the backtester's only trusted/registered strategy for the *overlay* engine path is `short_after_pump` (see Investigate below) — a submitted overlay bundle can never be evaluated against an arbitrary **submitted** baseline strategy, only against the curated one.

Native overlay-on-submitted-baseline would let the backtester run "baseline module + N overlay modules" as first-class inputs, closing both gaps at once and letting a future slice (G3b, LLM-consolidation) skip the lab-side codegen entirely.

## Investigate first (facts already verified in this repo — re-verify at implementation time, code moves)

- **One bundle slot on the wire.** `RunSubmitRequest` (`packages/sdk/src/contracts/run.ts:48`) carries exactly one optional bundle: `readonly moduleBundle?: ModuleBundle;`. There is no second slot for "the baseline this overlay applies to" — a submitted overlay and a submitted baseline cannot both ride the same request today.
- **The overlay engine path builds its registry with an empty `strategyBundles`.** `apps/backtester/src/jobs/worker.ts:514` (inside the `engine === 'overlay'` branch): `registry = buildInlineOverlayRegistry([sandboxBundle!.bundle]);` — only the first positional arg (`overlayBundles`) is passed. `buildInlineOverlayRegistry`'s signature (`apps/backtester/src/engine/trusted-registry.ts:29-31`) is `(overlayBundles: readonly ModuleBundle[], strategyBundles: readonly ModuleBundle[] = [])` — so on the overlay path, `strategyBundles` is always `[]`; the ONLY strategies in scope come from `TRUSTED_REGISTRY_DEFINITION.strategies`.
- **`TRUSTED_REGISTRY_DEFINITION.strategies` is `short_after_pump`-only.** `apps/backtester/src/engine/registry-definition.ts:35`: `strategies: [shortAfterPump]`. This is the single source feeding both `/v1/registry` discovery and inline overlay execution (`registry-execution-consistency.test.ts` guards the two never drift). A submitted overlay's `manifest.targetStrategyRef` can therefore only resolve against `short_after_pump` — never against a caller-submitted baseline.
- **Contrast: the `strategy` engine path DOES accept a submitted bundle as the "strategy".** `apps/backtester/src/jobs/worker.ts` (`engine === 'strategy'` branch): `const registry = buildInlineOverlayRegistry([], [sandboxBundle!.bundle]);` — here the submitted bundle rides `strategyBundles`, with `overlayBundles` empty. This is the exact mechanism G3's lab-side bridge already exploits (see below): a composed revision is submitted as a **whole, self-sufficient `kind:'strategy'` module**, not as an overlay-on-baseline pair.
- **`targetStrategyRef` is a real, enforced gate today, just scoped to the trusted catalog.** `packages/sdk/src/builder/manifest.ts:27` (`CreateModuleManifestInput.targetStrategyRef`) plumbs into `ModuleManifest.targetStrategyRef`; `apps/backtester/src/engine/runner.ts:753`'s 017 validation resolves `knownStrategyRefs` from the registry's trusted strategies (`registry-route.ts`, `overlay-sandbox-materialize.test.ts:39-41` confirm the same pattern for tests: known refs = the module itself + its `targetStrategyRef`). For overlay-on-submitted-baseline to work, this gate must accept a ref that resolves to the **newly-submitted** baseline bundle's `manifest.id`, not just the trusted catalog.

## Change (ask)

1. **A second bundle slot on the wire.** Add `readonly baselineModuleBundle?: ModuleBundle;` alongside `moduleBundle` on `RunSubmitRequest` (and `ModuleValidateRequest` if validate-before-submit should also support the pair). Additive, optional — existing single-bundle overlay and strategy submissions are unaffected.
2. **Worker wiring on the overlay path.** In `apps/backtester/src/jobs/worker.ts`'s `engine === 'overlay'` branch, when `baselineModuleBundle` is present on the claimed request, pass it through: `registry = buildInlineOverlayRegistry([sandboxBundle!.bundle], [baselineBundle])` — i.e. actually populate the `strategyBundles` param that is unconditionally `[]` today (see Investigate). When absent, behavior is byte-identical to today (falls back to `TRUSTED_REGISTRY_DEFINITION.strategies`, i.e. `short_after_pump`).
3. **`targetStrategyRef` / 017 `knownStrategyRefs` gate.** Extend the known-strategy-ref resolution (`runner.ts:753` and wherever `registry-route.ts`/validation derive it) so that when a `baselineModuleBundle` is submitted, its `manifest.id` is added to `knownStrategyRefs` for that request — the submitted overlay's `targetStrategyRef` should be allowed to point at the just-submitted baseline, not only at the trusted catalog. Keep the trusted-catalog refs always known (no regression for existing `short_after_pump`/`early_exit_short_after_pump` flows).
4. **Multi-overlay bundles (stretch, same ask family).** `buildInlineOverlayRegistry`'s first arg is already `readonly ModuleBundle[]` — i.e. the registry-building primitive is N-overlay-capable today. The wire contract is the actual bottleneck: `RunSubmitRequest.moduleBundle` is singular. If native overlay-on-submitted-baseline lands, consider whether `moduleBundle` should become `moduleBundle?: ModuleBundle` (singular, unchanged) + accept multiple overlay bundles some other way (array field, or N sequential submissions composed server-side) — no strong opinion from the lab side; flag as an open design question rather than a fixed ask.

## Acceptance

- `RunSubmitRequest`/`ModuleValidateRequest` accept an optional `baselineModuleBundle: ModuleBundle` alongside the existing `moduleBundle` — additive contract bump, one SDK release.
- Submitting `{ engine: 'overlay', moduleBundle: <overlay>, baselineModuleBundle: <baseline> }` runs the overlay against the **submitted** baseline (not `short_after_pump`) — `sandboxBundle!.bundle`'s `strategyBundles` slot in the worker's overlay-path registry actually gets populated, verified by a new `apps/backtester` integration test analogous to `registry-execution-consistency.test.ts`.
- The overlay's `manifest.targetStrategyRef` resolves against the submitted baseline's `manifest.id` (017 validation passes; `overlay-unknown-strategy`-style rejection still fires for refs that match neither the trusted catalog nor the submitted baseline).
- Existing single-bundle overlay (`short_after_pump` implicit baseline) and single-bundle strategy submissions are unaffected — no behavior change when `baselineModuleBundle` is absent.
- `@trading-backtester/sdk` release documents the new field + the `targetStrategyRef` resolution change in the changelog.

## NOT in scope

- No change to `OverlayComposer`'s veto/patch/annotate/pass semantics itself — this handoff is purely about *which baseline* an overlay runs against on the wire, not the composition algorithm (`compose-revision-bundle.ts`'s header already documents that the lab-side composer mirrors this semantics faithfully; native overlay-on-submitted-baseline does not obsolete that harness for G3, since G3 also needs deterministic conflict-detection + greedy degradation *before* any backtest run).
- No LLM-consolidation of a revision's composed source into a single clean module — that is G3b, a separate, unrelated slice gated on this handoff or not (independent).
- No change to `TRUSTED_REGISTRY_DEFINITION`'s curated catalog (`short_after_pump` stays the only *trusted-for-evidence* strategy) — this handoff is about accepting an *additional*, request-scoped, untrusted baseline for the duration of one run, not about growing the trusted catalog.
- No multi-overlay wire-contract change is committed to (see Change item 4 — flagged as an open question, not an ask).

## Lab-side contract (for reference — how G3 works today, and what changes if this lands)

**Today (shipped, works, no dependency on this handoff):** `StrategyRevisionRunExecutor` (`src/ports/strategy-revision-run-executor.ts`) is the seam G3's `revisionBuildHandler` (`src/orchestrator/handlers/revision-build.handler.ts`) calls through — `execute({ revisionId, label, strategyBundle, strategyProfileId, run, metrics, correlationId })`. Its only implementation, `BacktesterRevisionRunExecutor` (`src/research/backtester-revision-run-executor.ts`), submits the **already-composed** revision bundle (base + every included overlay, namespace-isolated by `composeRevisionBundle`) as a single `kind:'strategy'` module via the existing G1 strategy-lane (`engine:'strategy'`, which — per Investigate above — already accepts a submitted bundle through `strategyBundles`). No backtester change was needed to ship G3; this is the "shim" the spec's §5 note (*"Шов под будущий handoff... сейчас единственная реализация делегирует в G1 strategy-lane"*) refers to.

**If this handoff lands:** a second `StrategyRevisionRunExecutor` implementation could submit `{ engine: 'overlay', moduleBundle: <overlay-only>, baselineModuleBundle: <latest accepted revision's bundle> }` per overlay (or per batch, depending on how multi-overlay is eventually resolved — see Change item 4), without composing lab-side at all. `revisionBuildHandler` itself would not need to change — the port boundary already isolates this decision, by design (spec §5).
