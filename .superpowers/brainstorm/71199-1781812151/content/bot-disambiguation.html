Уточнение раздела 3. Как разрешать ссылку на бота

«Бот» — отдельная сущность. Сначала выбираем botId, затем читаем его runs и результаты.

Ранжирование кандидатов

exact ID/name → alias/name prefix → strategy attributes → active/recent run → lastBotId из сессии как проверяемая подсказка

LLM извлекает ограничения из фразы. Список и порядок формирует детерминированный resolver; сессия никогда не считается доказательством существования бота.

Состояние диалога

pendingInteraction: { kind: 'entity_disambiguation', entityType: 'bot', query, candidateIds, cursor, expiresAt }

Ответ «2» выбирает snapshot-кандидата без нового LLM-вызова. «Нет в списке» переводит диалог в уточнение; новый запрос заменяет старый список.

Новый read-контракт

  • Каталог botId, displayName, aliases, strategyRef, market/timeframe, status.
  • После выбора — runs и summaries через BotResultsReadPort.
  • Человеческий label всегда содержит различающие признаки.

Чего не делаем

  • Не показываем сотни ботов одним списком.
  • Не выбираем по одному слову при нескольких кандидатах.
  • Не запоминаем номер «2» вне конкретного candidate snapshot.
