Раздел 1. Архитектура и state machine

Подтверждается не intent, а конкретный неизменяемый Action Proposal.

Один пользовательский цикл
1. Understand
Subject + requested goal
LLM, без side effects
→
2. Enrich
DB + ports + similarity + RAG
только чтение
→
3. Propose
План, факты, альтернативы
сохраняется в session

↓

WAITING_CONFIRMATION
Никакой task ещё не существует

↓

Confirm
Guard создаёт задачу
Alternative
Новое proposal
Edit
Уточнить subject/goal
Cancel/Expire
Закрыть без действий

Границы компонентов

Turn Interpreter
Определяет предмет, цель и ссылки из текущего сообщения + session context.
Evidence Enricher
Собирает типизированные факты. Не генерирует выводы и ничего не пишет.
Proposal Policy
Детерминированно выбирает допустимые действия и формирует Action Proposal.
Confirmation Guard
Принимает только ответ на актуальный proposal и вызывает существующий task intake.

Ключевой контракт

ActionProposal { id, subjectSnapshot, evidenceRefs, action, payloadPreview, alternatives, expiresAt, status }

Payload вычисляется до вопроса. При подтверждении guard проверяет proposal ID, срок, статус и неизменность subject; затем использует сохранённый payload, а не заново интерпретирует «да» через LLM.

Гарантии

  • Любая новая стратегия сначала становится proposal.
  • Даже «проанализируй» не обходит confirmation.
  • Короткое «да» связано с одним proposal ID.

Не считаем подтверждением

  • Низкоуверенный ответ.
  • Ответ после истечения proposal.
  • Новое описание стратегии вместо явного согласия.
